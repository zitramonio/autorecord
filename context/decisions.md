# Решения

- 2026-05-06: Создана папка `context/` для хранения проектного контекста.
  - Причина: пользователь запросил структуру для фиксации состояния проекта, задач и решений.

- 2026-05-06: MVP будет Windows-only приложением на .NET WPF с NAudio.
  - Причина: нужен GUI, запись системного вывода и микрофона, фонова работа и автозапуск после перезапуска компьютера.

- 2026-05-06: Встречи читаются из Яндекс.Календаря через приватную iCal-ссылку.
  - Причина: этого достаточно для чтения событий и проще, чем авторизация через аккаунт.

- 2026-05-06: События календаря обновляются раз в час и вручную по кнопке в GUI.
  - Причина: пользователь подтвердил такой режим обновления.

- 2026-05-06: Микрофон и системный вывод сохраняются в один общий файл.
  - Причина: пользователю нужен один файл записи встречи.

- 2026-05-06: После настраиваемого периода тишины приложение спрашивает, остановить ли запись.
  - Причина: пользователь хочет вручную подтверждать остановку; при ответе `Нет` повторная проверка выполняется через настраиваемый интервал.

- 2026-05-06: Для реализации подготовлен пошаговый план `docs/superpowers/plans/2026-05-06-autorecord-mvp.md`.
  - Причина: спецификация утверждена, следующий этап — реализация по проверяемым задачам.

- 2026-05-06: Реализация выполнена в отдельном worktree `.worktrees/mvp` на ветке `feature/mvp`.
  - Причина: безопасная изоляция разработки от основного дерева проекта.

- 2026-05-06: .NET SDK 10 установлен локально в `C:\Users\User\.dotnet`.
  - Причина: системный `dotnet` не был доступен в `PATH`, а проект требует .NET 10.

- 2026-05-06: В recorder реализовано смешивание микрофона и системного вывода в общий WAV.
  - Причина: это ключевое требование MVP; промежуточный вариант с записью только loopback был отклонен review.

- 2026-05-06: Lifecycle записи защищен от гонок start/stop, ошибок старта/остановки и повторных stop prompt.
  - Причина: фоновые аудио callbacks, GUI и shutdown могут выполняться параллельно.

- 2026-05-06: Ручное обновление календаря теперь использует текущие значения формы, а не только сохраненные настройки.
  - Причина: до `fbfb519` кнопка `Обновить календарь` могла отправлять пустой URL, если пользователь вставил iCal-ссылку, но еще не нажал `Сохранить`.

- 2026-05-07: Для запуска без PowerShell выбран publish в папку с обычным `Autorecord.App.exe`, а не single-file и не инсталлятор.
  - Причина: для WPF это самый надежный и простой вариант; `.exe` запускается двойным кликом вместе с соседними publish-файлами.

- 2026-05-07: В GUI добавлены статус записи и ручные кнопки старта/остановки.
  - Причина: пользователь хочет видеть состояние записи и управлять записью вручную из приложения; доступность кнопок должна строго зависеть от активной записи.

- 2026-05-07: Для защиты начала ручной записи используется `KeepMicrophoneReady` без добавления pre-roll в файл.
  - Причина: диагностическая задержка подтвердила проблему прогрева микрофона, но pre-roll добавлял в новую запись фрагменты звука до нажатия кнопки.

- 2026-05-07: Recorder держит аудиоустройства подготовленными только при включенной настройке `Держать микрофон готовым для мгновенного старта`.
  - Причина: это снижает риск влияния на другие приложения/устройства и оставляет пользователю контроль над компромиссом.

- 2026-05-07: Календарный polling ускорен до 1 секунды, встреча может стартовать на 3 секунды раньше.
  - Причина: нужно компенсировать задержку аудиоустройств и точнее попадать в начало встречи.

- 2026-05-07: Итоговый формат записей изменен на `.mp3`, внутри используется временный 16-bit PCM WAV.
  - Причина: пользователю нужен меньший итоговый файл; временный WAV надежнее для живого микширования и в 16-bit PCM занимает примерно вдвое меньше прежнего float WAV.

- 2026-05-07: MP3 кодируется через MediaFoundation с битрейтом 128 kbps.
  - Причина: это доступно через NAudio/Windows без отдельного ffmpeg и достаточно для записи встреч.

- 2026-05-07: Звук до нажатия `Начать запись` не должен попадать в итоговый файл.
  - Причина: пользователь ожидает, что ручная запись начинается с момента нажатия кнопки, даже если микрофон заранее подготовлен.

- 2026-05-07: MP3 сохраняется в фоне после быстрого закрытия временного `.recording.wav`; WAV удаляется только после успешного финального `.mp3`.
  - Причина: остановка записи не должна блокироваться долгой MP3-конвертацией, но потеря записи критична, поэтому временный WAV остается резервной копией до подтвержденного успеха.

- 2026-05-07: Финальный `.mp3` публикуется только через временные файлы, а незавершенные `.recording.wav` восстанавливаются при запуске.
  - Причина: нужно снизить риск поврежденного/частично сохраненного финального файла и иметь путь восстановления после падения приложения или выключения компьютера.

- 2026-05-07: v2 строится как локальная транскрибация поверх текущей записи без облачных ASR API, телеметрии и отправки аудио/транскриптов наружу.
  - Причина: пользователь требует полностью локальный offline-сценарий после скачивания моделей.

- 2026-05-07: Текущий финальный формат записи `.mp3` не меняется; транскрибация принимает сохраненный аудиофайл и создает временный normalized `.wav` внутри pipeline.
  - Причина: это минимизирует риск сломать уже рабочее MP3-сохранение и safe-save поведение.

- 2026-05-07: Папка аудиозаписей и папка транскриптов проектируются как разные настройки.
  - Причина: пользователь хочет иметь возможность хранить транскрипты отдельно от записей.

- 2026-05-07: Диаризация остается отдельным модулем от ASR, но входит в единый автоматический pipeline транскрибации после записи.
  - Причина: нужно получать готовые транскрипты со спикерами автоматически, сохраняя расширяемость для разных ASR-моделей.

- 2026-05-07: Реализация v2 планируется фазами: foundation/UI, модели/скачивание, очередь/экспорт, sherpa ASR, диаризация, GigaAM v3, интеграция после записи.
  - Причина: sherpa-onnx имеет .NET/C# путь и подходит для первого рабочего local-ASR, а GigaAM v3 требует более рискованного bundled worker/backend решения.

- 2026-05-07: Для v2 создан отдельный implementation plan `docs/superpowers/plans/2026-05-07-autorecord-v2-transcription.md`.
  - Причина: scope включает несколько подсистем, поэтому разработку нужно вести проверяемыми фазами A-G с тестами и checkpoint-ами.

- 2026-05-07: v2 исполняется в режиме Subagent-Driven с обязательными spec-review и code-quality review после каждой задачи.
  - Причина: scope большой и рискованный; review gates уже выявили и закрыли важные edge cases в JSON/path/file safety.

- 2026-05-07: Для локальных JSON-хранилищ и transcript-файлов выбран паттерн записи через временный файл и последующий move/replace.
  - Причина: нельзя терять очередь, историю или транскрипт при отмене, падении процесса или ошибке записи.

- 2026-05-07: Для ModelManager и ModelDownloadService введены path-safety ограничения внутри выделенных root-папок.
  - Причина: catalog/model id/required files не должны позволять удалять или проверять файлы вне `%LOCALAPPDATA%\Autorecord\Models` и download root.

- 2026-05-07: В `TranscriptionQueue` разделены lock-и для списка jobs, repository save и single-worker pipeline.
  - Причина: транскрибация может идти долго, но GUI должен иметь возможность добавлять новые задания в очередь без ожидания завершения текущего pipeline.

- 2026-05-07: Cleanup временного normalized WAV не выполняется в `AudioNormalizer`; он только возвращает `CreatedTemporaryFile`.
  - Причина: решение об удалении промежуточных файлов зависит от pipeline-настроек `KeepIntermediateFiles` и должно приниматься на уровне pipeline.

- 2026-05-07: Для первого ASR-движка выбран `org.k2fsa.sherpa.onnx` 1.13.0 и offline NeMo CTC config.
  - Причина: это локальный .NET/C# путь для модели `sherpa-gigaam-v2-ru-fast` без Python, Docker и облачных API.

- 2026-05-07: `SherpaOnnxTranscriptionEngine` временно возвращает один full-duration segment и читает WAV целиком.
  - Причина: модельный decode подключен в Task 10, а точный chunking/timestamps для длинных записей должен быть вынесен в следующий pipeline/chunking этап.

- 2026-05-07: Speaker IDs в `DiarizationEngine` нормализуются в формат `SPEAKER_00`.
  - Причина: это совпадает с JSON/exporter-планом v2 и дает стабильную основу для будущего `TranscriptAssembler` и отображения `Speaker 1`, `Speaker 2`.

- 2026-05-07: ASR и diarization на sherpa-onnx пока читают normalized WAV целиком.
  - Причина: Task 10/11 фиксируют рабочую runtime-интеграцию; chunking/streamed read для длинных записей переносится в pipeline/chunking этап.

- 2026-05-08: `TranscriptAssembler` закрепляет стабильные speaker labels: `SPEAKER_00 -> Speaker 1`, `SPEAKER_01 -> Speaker 2`, fallback всегда `SPEAKER_00 / Speaker 1`.
  - Причина: transcript output не должен менять label для одного speaker id и не должен создавать duplicate labels между fallback и нестандартными speaker ids.

- 2026-05-08: GigaAM v3 реализован как локальный worker contract/shell, а не как встроенная модельная runtime-интеграция.
  - Причина: это сохраняет требование без командной строки для пользователя и оставляет реальную упаковку worker/dependencies отдельным этапом без облачных API.

- 2026-05-08: `TranscriptionPipeline` валидирует тип и установленность ASR/diarization моделей до нормализации аудио.
  - Причина: ошибки выбора или отсутствия модели должны возникать fail-fast, без дорогой обработки аудио и без лишних временных файлов.

- 2026-05-08: Progress в `TranscriptionPipeline` маппится по стадиям и не должен откатываться назад.
  - Причина: очередь и GUI не должны видеть последовательность вроде `100 -> 0` между диаризацией и ASR.

- 2026-05-08: `TranscriptionQueue` в WPF-приложении должна быть долгоживущей, а pipeline создаваться свежим под текущие настройки.
  - Причина: нельзя запускать параллельные очереди или перетирать snapshot repository, но смена настроек должна влиять на следующие задания.

- 2026-05-08: Для live progress в GUI добавлен `TranscriptionQueue.JobsChanged` без сохранения каждого progress tick в repository.
  - Причина: UI должен видеть `Running`/progress/terminal states, а `jobs.json` не должен получать чрезмерную частоту записи.

- 2026-05-08: GUI-download моделей пока должен быть честно ограничен скачиванием temp-файла до реализации install/extract/manifest.
  - Причина: текущий `ModelDownloadService` еще не выполняет полноценную установку модели и проверку installed manifest.

- 2026-05-08: Автотранскрибация подключается только после события `RecordingSaved`, а не после остановки записи.
  - Причина: в очередь должен попадать уже сохраненный финальный `.mp3`, чтобы не ломать safe-save и не обрабатывать временный `.recording.wav`.

- 2026-05-08: Логика enqueue после записи вынесена в небольшой `RecordingTranscriptionEnqueuer`.
  - Причина: нужно покрыть автотранскрибацию unit-тестами без WPF-host и сохранить единый выбор output directory/ASR/diarization параметров.

- 2026-05-08: Полноценная установка моделей вынесена в `ModelInstallService`, а `ModelDownloadService` оставлен сетевым/temp компонентом.
  - Причина: сетевой download, локальная распаковка/проверка/manifest и GUI orchestration имеют разные границы ответственности.

- 2026-05-08: Для `tar.bz2` моделей добавлен `SharpCompress`.
  - Причина: .NET умеет TAR, но не дает встроенного BZip2 decompressor; sherpa-модели поставляются как `.tar.bz2`.

- 2026-05-08: Установка модели публикует target-папку через staging и держит backup до успешной записи `manifest.json`.
  - Причина: при сбое или отмене manifest update после publish нельзя терять уже установленную старую модель или оставлять несогласованное состояние.

- 2026-05-08: Для diarization-моделей установка рассматривает segmentation archive и embedding `.onnx` как artifacts одной модели.
  - Причина: GUI должен установить модель диаризации одним действием, а `ModelManager` должен видеть все `requiredFiles` в одном target folder.

- 2026-05-08: В GUI `null` selection диаризации трактуется как отсутствие готового UI selection и сохраняет текущие настройки; пустой sentinel `""` означает явный выбор `Без разделения по спикерам`.
  - Причина: WPF ComboBox может временно иметь `SelectedItem = null`, и это не должно молча отключать уже сохраненную диаризацию.

- 2026-05-08: При GUI-download моделей приложение сначала читает текущие значения формы, даже если пользователь еще не нажал `Сохранить`.
  - Причина: скачивание должно соответствовать выбранной прямо сейчас ASR/diarization конфигурации, а не устаревшему snapshot настроек.

- 2026-05-08: Действия истории транскрибации реализованы на уровне `TranscriptionQueue`, а WPF передает только `JobId`.
  - Причина: retry/cancel/delete должны сохранять локальную очередь и историю консистентно, независимо от UI.

- 2026-05-08: Running transcription job обновляется по `JobId`, а не по сохраненному индексу списка.
  - Причина: удаление старой history-строки во время выполнения задачи может сместить индексы и привести к обновлению неверной задачи.

- 2026-05-08: Persistent save очереди берет актуальный `_jobs` snapshot внутри `SaveCurrentAsync`, а не пишет заранее захваченную копию.
  - Причина: action-сохранение не должно откатывать более свежий terminal transition running job.

- 2026-05-08: Во время active download/install модели GUI блокирует `Скачать`, `Удалить` и `Проверить`, оставляя доступной только отмену.
  - Причина: model install/delete/validate используют одну локальную папку моделей и manifest; параллельные действия могут создать гонку и неконсистентный статус.

- 2026-05-08: `LocalNetworkGuard` защищает transcription runtime от сетевых API, а `ModelDownloadService` остается единственной сетевой boundary-точкой внутри transcription-кода.
  - Причина: v2 допускает интернет только для явного скачивания моделей через GUI; ASR, диаризация, pipeline, очередь и экспорт должны работать локально и не отправлять аудио/транскрипты наружу.

- 2026-05-08: Логи заданий транскрибации пишутся локально best-effort и не влияют на terminal status задания.
  - Причина: job-log нужен для диагностики v2, но ошибка записи лога или чтения duration из поврежденного `.wav` не должна ломать очередь, запись или сохранение транскрипта.

- 2026-05-08: Автотранскрибация включена по умолчанию для новых/старых настроек без секции `Transcription`.
  - Причина: основной UX v2 — после сохранения записи файл автоматически попадает в локальную очередь транскрибации; старые `settings.json` MVP не должны молча отключать этот сценарий.

- 2026-05-08: Пустой результат ASR не считается ошибкой транскрибации.
  - Причина: тихая запись или файл без распознанной речи должны завершать job штатно и создавать transcript-файлы с понятным сообщением `Речь не обнаружена.`, а не падать с технической ошибкой валидации.

- 2026-05-08: Для длинных записей sherpa ASR обрабатывает normalized WAV фиксированными chunks по 20 секунд с 0.25 сек padding.
  - Причина: 1-2 часовые записи не должны читаться целиком в память; при этом ASR получает небольшой контекст на границах, а transcript сохраняет глобальные таймкоды.

- 2026-05-08: Full-chunk дедупликация текста на границах ASR chunks не используется.
  - Причина: сравнение полного текста соседних chunks может удалить валидный повтор речи; точную boundary-dedupe нужно делать отдельным этапом по более надежным данным.

- 2026-05-08: Для длинных записей diarization обрабатывает normalized WAV chunks по 30 минут с 5 сек padding.
  - Причина: 1-2 часовые записи не должны читаться целиком в память, но диаризации нужен более длинный контекст, чем ASR.

- 2026-05-08: Speaker ids между diarization chunks сопоставляются только по реальному overlap в padding.
  - Причина: cluster ids sherpa diarization локальны для каждого chunk; нельзя слепо считать `SPEAKER_00` в разных chunks одним человеком или маппить speakers через искусственные gaps после normalization.

- 2026-05-09: GigaAM v3 worker source и packaging hook хранятся в репозитории, но готовый `worker.exe`, Python runtime и тяжелые зависимости не коммитятся.
  - Причина: приложению нужен bundled локальный backend без командной строки для пользователя, но бинарные артефакты должны собираться отдельно и подключаться через `artifacts/vendor/gigaam-worker`.

- 2026-05-09: Для `gigaam-v3-ru-quality` каталог моделей скачивает два plain artifacts: `v3_e2e_rnnt.ckpt` и `v3_e2e_rnnt_tokenizer.model`.
  - Причина: worker должен работать offline после GUI-скачивания модели и проверять наличие required files до загрузки GigaAM runtime.

- 2026-05-09: GigaAM worker обходит upstream `model.transcribe(file)` и не зависит от `ffmpeg` в runtime.
  - Причина: upstream GigaAM читает audio file через внешний `ffmpeg`, а приложение не должно требовать от пользователя ручной установки ffmpeg; normalized WAV уже подготовлен pipeline-ом.

- 2026-05-09: Publish должен падать, если `artifacts/vendor/gigaam-worker/worker.exe` отсутствует.
  - Причина: нельзя выпускать приложение с доступной моделью `gigaam-v3-ru-quality`, но без bundled worker, иначе пользователь не сможет запустить quality mode без командной строки.

- 2026-05-09: Кнопка `Скачать` устанавливает только отсутствующие выбранные модели.
  - Причина: если ASR-модель уже установлена, а пользователь включил новую diarization-модель, приложение не должно повторно скачивать тяжелую ASR-модель; интернет разрешен только для явного и необходимого скачивания моделей.

- 2026-05-09: Установка multi-artifact модели может flatten-ить foldered archive поверх staging root с plain artifacts только после preflight conflict check.
  - Причина: diarization-модель состоит из segmentation `tar.bz2` с top-level папкой и plain embedding `.onnx`; required files должны оказаться в корне установленной модели, но partial move при конфликте недопустим.

- 2026-05-09: Точная дедупликация ASR boundary-overlap выполняется в `TranscriptAssembler` при merge соседних сегментов одного speaker.
  - Причина: это общий слой после ASR и диаризации, поэтому улучшение одинаково работает для Sherpa и GigaAM и не вмешивается в модельные движки; удаляются только точные совпадения suffix/prefix от двух слов и больше, чтобы снизить риск удаления валидных повторов речи.

- 2026-05-09: Автозапуск сохраняет Task Scheduler как основной механизм и использует `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` как fallback при `Access denied` / `E_ACCESSDENIED`.
  - Причина: Task Scheduler лучше подходит для фонового запуска `--minimized`, но в текущей Windows-сессии создание задачи блокируется политикой/правами; пользовательский чекбокс автозапуска должен работать без командной строки и админских действий.

- 2026-05-09: Отсутствующая ASR/diarization модель переводит задачу транскрибации в `WaitingForModel`, а не в `Failed`.
  - Причина: это восстановимое состояние пользователя: после скачивания модели через GUI задача должна продолжиться без ручного пересоздания и без ложной ошибки транскрибации.

- 2026-05-09: При старте приложения восстановленные `Pending` задачи транскрибации запускаются автоматически.
  - Причина: очередь хранится между перезапусками, поэтому незавершенные задания должны продолжать обработку без ручного нажатия, если модели уже доступны.

- 2026-05-09: GUI-действия `Удалить выбранные` и `Проверить выбранные` применяются к выбранному набору моделей, а не только к ASR.
  - Причина: пользователь выбирает ASR и режим диаризации как связанный runtime-набор; если включена diarization-модель, управление моделью должно включать и её.

- 2026-05-09: Пользовательские ошибки транскрибации и установки моделей маппятся через `UserFacingErrorMessages`.
  - Причина: в GUI не должны попадать сырые технические исключения; пользователю нужны понятные действия: скачать модель, проверить интернет/место, выбрать другую папку или повторить без неподдержанного формата.

- 2026-05-09: Timeout диалога остановки после молчания трактуется как подтверждение остановки.
  - Причина: если запись уже считается тихой и пользователь не отвечает 2 минуты, приложение должно завершить запись без зависания на модальном окне и пройти обычный path сохранения/автотранскрибации.

- 2026-05-09: `SharpCompress` обновлен до `0.48.0` в рамках release hardening.
  - Причина: версия `0.40.0` давала vulnerability warning `NU1902`; перед v2-релизом known vulnerable dependency не оставляем.

- 2026-05-11: Recorder writer выравнивается по wall-clock timeline, а не по длительности итераций writer loop.
  - Причина: на длинной записи фиксированный цикл `20 ms + delay` накапливал drift, и итоговый MP3 мог быть короче фактической записи.

- 2026-05-11: System-output capture создается для всех активных render devices.
  - Причина: встреча может звучать в наушниках/гарнитуре или отдельном communications endpoint; захват только одного default loopback может пропустить звук собеседников.

- 2026-05-11: Pyannote Community-1 добавляется как отдельный diarization engine и локальный worker, а не как sherpa-onnx модель.
  - Причина: Pyannote Community-1 имеет другой runtime stack и Hugging Face snapshot-формат; смешивание с текущим sherpa-onnx `DiarizationEngine` дало бы неверный runtime path.

- 2026-05-11: Hugging Face token для Pyannote Community-1 запрашивается только в момент явного GUI-скачивания и не сохраняется.
  - Причина: модель gated и требует принятия условий/token для скачивания, но приложение не должно хранить секреты без отдельного решения и не должно делать сетевые запросы во время транскрибации.

- 2026-05-11: Pyannote worker явно отключает Hugging Face/pyannote telemetry и запускается в offline-mode.
  - Причина: требования v2 запрещают телеметрию и любые сетевые операции во время ASR/диаризации; интернет разрешен только для явного скачивания моделей.

- 2026-05-11: В истории заданий нужен отдельный столбец `Модель диаризации`.
  - Причина: пользователь тестирует качество разных diarization-моделей, поэтому рядом с ASR-моделью должно быть видно, какой diarization backend использовался.

- 2026-05-11: Pyannote Community-1 worker не должен читать аудио через `torchaudio/torchcodec`; он читает normalized WAV стандартным Python `wave` и передает Pyannote in-memory tensor.
  - Причина: приложение уже нормализует аудио до PCM WAV, а `torchcodec/libtorchcodec` на Windows требует отдельную FFmpeg/runtime-совместимость; пользователь не должен ставить FFmpeg, Python-пакеты или чинить worker вручную.

- 2026-05-12: При включенной диаризации ASR должен транскрибировать интервалы diarization turns, а не весь файл крупными ASR chunks.
  - Причина: без word-level timestamps невозможно корректно разделить 20-секундный ASR chunk, если внутри него говорят разные люди; назначение одного speaker всему chunk смешивало реплики мужского и женского голосов.

- 2026-05-12: GigaAM worker получает интервалы диаризации через локальный `--chunks-json`.
  - Причина: GigaAM должен работать offline и без командной строки, но при diarization-on обязан транскрибировать уже нарезанные speaker turns, чтобы итоговый transcript не объединял реплики разных спикеров.

- 2026-05-12: Для релизного UX v2 оставлены только модели `gigaam-v3-ru-quality` и `pyannote-community-1`; выбор моделей в GUI убран.
  - Причина: текущий продуктовый сценарий должен быть простым и предсказуемым: качественная русская транскрибация + лучшая доступная диаризация без тестового выбора старых моделей пользователем.

- 2026-05-12: Дефолтная папка записей изменена на `Documents\Autorecord`.
  - Причина: записи и транскрипты не должны складываться напрямую в корень `Documents`; новые/пустые настройки получают отдельную папку приложения.

- 2026-05-12: При остановке записи появляется отдельный prompt выбора числа спикеров `Auto/1..6`, но запись останавливается и сохраняется сразу.
  - Причина: ожидание выбора не должно продлевать аудиозапись; выбранное число спикеров влияет только на параметры последующей локальной транскрибации/диаризации. Если выбора нет 2 минуты, используется `Auto`.

- 2026-05-12: История транскрибаций убрана из релизного GUI; показывается только текущая/последняя актуальная задача.
  - Причина: для релиза пользователю нужен прогресс текущей обработки, а не накопительная таблица тестовых заданий; старые задачи чистятся до актуальной задачи.

- 2026-05-12: Блок управления моделями в `Транскрибация` скрывается, когда GigaAM v3 и Pyannote Community-1 уже установлены.
  - Причина: после первичной установки этот блок не нужен в основном UX; он остается видимым только если модель отсутствует, идет скачивание или требуется восстановление/проверка.
